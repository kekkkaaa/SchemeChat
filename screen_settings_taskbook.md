# Screen Settings Taskbook

## 目标
在现有 `Setting` 按钮弹窗内，增加一套可执行的屏幕排版设置能力：

- 弹窗左侧是 tab 列表
- 第一个 tab 为 `Screen`
- 右侧显示当前屏幕容器排版详情
- 支持调整容器数量
- 支持 3 种布局模式
- 支持右侧预览
- 支持点击 `Apply` 后应用到主界面

这份文档是后续实现的直接执行书。

---

## 当前基线

当前项目仍然是固定双屏结构，核心特征如下：

- 主窗口网页容器是写死的 `left / right`
- 布局计算也是按双栏写死
- `Setting` 弹窗目前只是空壳
- 没有动态 pane 列表
- 没有布局引擎
- 没有屏幕设置状态模型

结论：
这次不是只改弹窗 UI，而是要把“固定双栏窗口管理”升级为“动态容器列表 + 布局计算 + 设置应用”。

---

## 本次范围

本次只做最小可用版，不做过度设计。

要做：

- `Setting` 弹窗改为左右结构
- 左侧 tab 列表
- 右侧 `Screen` 页
- 容器数量调节
- 3 种布局模式
- 布局预览
- 应用到主界面

暂时不做：

- 容器分组
- 多屏幕显示器适配

- 每个容器独立高级配置面板

## 容器数量规则

### 目标原则

容器数量在架构上不设上限，不写死 2、4、6。

### 现实约束

虽然代码层面不封顶，但真实可开数量仍受这些因素限制：

- Electron WebContents 数量
- AI 官方网页本身内存占用
- CPU / GPU 渲染压力
- 登录态和页面稳定性

所以本次实现规则是：

- 数据结构不封顶
- UI 调节允许持续增加
- 实际体验上由机器性能决定上限

---

## 3 种布局模式

第一版只做 3 种最实用模式。

### 1. Grid

规则：

- 自动按接近正方形的方式分配网格
- 适合大多数容器数量

示例：

- 1 个：1x1
- 2 个：2 列 1 行
- 3-4 个：2x2
- 5-6 个：3x2

### 2. Columns

规则：

- 所有容器按列横向平铺
- 每个容器等宽
- 高度占满网页区域

适合对比多个网页同时展开。

### 3. Rows

规则：

- 所有容器按行纵向平铺
- 每个容器等高
- 宽度占满网页区域

适合纵向阅读或低宽度场景。

## 主窗口数据模型

当前固定模型：

- `POSITIONS = ['left', 'right']`

目标模型：

```js
panes = [
  { id: 'pane-1', providerKey: 'chatgpt' },
  { id: 'pane-2', providerKey: 'gemini' }
]
```

### 必要状态

- `panes[]`
- `layoutMode`
- `paneCount`
- `supersizedPaneId | null`

### 原则

- 容器是数组，不再是固定位置名
- 布局函数只消费 `panes + layoutMode + windowBounds`
- provider 切换逻辑改成按 `pane.id` 处理

---

## 布局引擎职责

新增统一布局计算层，输出每个 pane 的 bounds。

建议接口：

```ts
computePaneBounds({
  panes,
  layoutMode,
  contentBounds,
  supersizedPaneId
})
```

### 输出

返回：

- 每个 pane 的 `x`
- `y`
- `width`
- `height`

### 要求

- 只负责算布局
- 不处理 provider DOM
- 不处理弹窗 UI
- 方便以后继续加新布局模式

---

## 预览区要求

`Screen` 页右侧必须有一个当前布局预览。

### 预览内容

- 当前容器数量
- 当前布局模式
- 每个容器的相对位置
- 容器编号

### 预览原则

- 预览不承载真实网页
- 只用简化块状模型展示
- 调节数量和布局时实时变化

---

## Apply 行为定义

点击 `Apply` 后：

1. 读取当前设置弹窗里的 `paneCount`
2. 读取 `layoutMode`
3. 主进程更新 pane 配置
4. 如果容器数量增加：
   - 创建新增 pane
   - 给新增 pane 分配默认 provider
5. 如果容器数量减少：
   - 销毁超出数量的 pane
   - 保留前面的 pane
6. 调用布局引擎重新计算
7. 主界面立即重排

### 默认 provider 规则

建议第一版这样处理：

- 前两个容器默认保留当前逻辑：`chatgpt` / `gemini`
- 新增容器默认复用最后一个 provider，或统一给 `chatgpt`

实现时只要求稳定，不要求完美。

## 配置持久化建议

第一版建议只持久化最关键的内容：

```json
{
  "paneCount": 2,
  "layoutMode": "grid",
  "panes": [
    { "id": "pane-1", "providerKey": "chatgpt" },
    { "id": "pane-2", "providerKey": "gemini" }
  ]
}
```

建议放在：

- `config/window-layout.json`

目的：

- 应用重启后恢复上次布局
- 避免每次都回到固定双栏

## 关键风险

### 风险 1：旧代码大量依赖固定位置名

当前很多逻辑直接假设只有 `left/right`。

解决方向：

- 逐步把位置键改为 `pane.id`
- 布局、provider、IPC 都改成数组驱动

### 风险 2：多个网页容器会显著增加资源消耗

解决方向：

- 架构不封顶
- 体验上允许用户自己控制数量
- 第一版不强加复杂限制

### 风险 3：现有 supersize 逻辑和动态 pane 可能冲突

解决方向：

- 先保留单一 `supersizedPaneId`
- 由布局引擎统一处理 supersize 分支

本任务不是“给设置弹窗加一个页面”这么简单。

- 主进程层：动态 pane 管理

- 配置层：最小持久化

后续执行时，必须严格按本任务书的阶段顺序推进，先让结构成立，再让布局落地。
