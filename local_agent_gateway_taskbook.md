# 本地 Agent 执行网关任务书

## 1. 文档目的

本文档定义 SchemeChat 后续可扩展的一个明确方向：

- 不做自建云端服务
- 不把网页控制权交给远程浏览器
- 只在用户本机提供一层受控 API / Tool 接口
- 让运行在本机的 Agent 宿主调用这层接口
- 由本机 Electron 客户端继续控制 ChatGPT / Gemini / 其他网页

这份文档不是泛泛而谈的想法记录，而是可以直接指导后续实现的任务书。

---

## 2. 核心结论

### 2.1 这个方向是可行的

可行，但前提必须说清楚：

- 不是“云端模型直接访问本机 API”
- 不是“浏览器里的网页聊天机器人直接访问 localhost”
- 而是“运行在用户本机的 Agent 宿主程序，通过 tools / cmd / MCP / 本地 HTTP 调用本机接口”

典型宿主示例：

- Codex 桌面端 / 本地工具调用环境
- OpenClaw 一类可调用本地命令或本地服务的 Agent 宿主
- IDE 内置 agent
- 未来可接的本地 MCP Client

### 2.2 这个项目未来的正确定位

SchemeChat 不应该演进成“云端多 Agent 平台”，而应该演进成：

**本地网页执行器 + 本地 Agent 执行网关**

也就是：

- Agent 负责思考、规划、决定下一步动作
- SchemeChat 负责在本机网页里输入、发送、等待、抓取、同步
- 一旦遇到登录失效、人机验证、异常页面，SchemeChat 进入暂停状态，等人处理

---

## 3. 为什么不做纯远程

如果后期强行走纯远程网页代理，会遇到以下硬问题：

1. 网页登录态不稳定  
   ChatGPT、Gemini、Grok 这类页面都可能频繁要求重新登录或校验。

2. 人机验证无法自动闭环  
   只要走网页，就必须接受“人工接管”是系统能力的一部分，而不是异常情况。

3. 云端 IP 更容易触发风控  
   远程浏览器、云主机、机房 IP 更容易触发验证码、二次确认、风控重试。

4. 用户本机已有自然登录环境  
   用户平时就在本机浏览器里登录，复用本机网页环境是最现实的路线。

所以本项目更适合：

- 网页运行在本机
- 登录态保留在本机
- Agent 能力也通过本机工具链接入

---

## 4. 目标架构

## 4.1 四层结构

### A. 用户层

用户仍然通过熟悉的界面操作：

- 在 SchemeChat 底部输入框中输入任务
- 查看多屏网页回复
- 在关键节点做人机验证
- 在高风险动作前点击确认

### B. Agent 宿主层

本机运行的 Agent 宿主负责：

- 接收用户目标
- 规划步骤
- 调用本地接口
- 读取当前状态
- 决定是否发送、是否同步、是否继续下一轮

### C. 本地执行网关层

这是本项目未来新增的一层能力，负责对外暴露有限的高层接口，例如：

- 设置输入文本
- 发送
- 同步讨论草稿
- 读取最新回复
- 查询当前状态
- 请求执行确认

这一层不应该直接暴露 DOM 细节。

### D. SchemeChat 网页执行层

现有 Electron 主体继续保留，负责：

- 控制 ChatGPT / Gemini 网页
- 把文本注入当前页面输入框
- 触发发送
- 检测是否仍在回复中
- 抓取最新回复
- 处理 Sync 逻辑
- 在人机验证时暂停自动流程

---

## 5. 设计原则

### 原则 1：只暴露高层语义动作

不要对外暴露：

- 点击第几个按钮
- 操作哪一个 DOM selector
- 页面坐标点击

应暴露：

- `set_input`
- `send`
- `sync_discussion_draft`
- `get_latest_replies`
- `wait_until_idle`
- `get_status`

这样后面网页结构变化时，Agent 不需要跟着改。

### 原则 2：本地优先，不依赖公网回调

系统不依赖你自己部署的服务器。

只要本机上同时存在：

- SchemeChat
- Agent 宿主

就可以跑通。

### 原则 3：状态机必须收口到 SchemeChat

真正的执行状态不应该由 Agent 自己胡乱猜，而应该由 SchemeChat 统一维护并返回。

### 原则 4：验证码不是异常，是正式状态

只要网页方案继续存在，就必须把以下状态作为一等公民：

- `requires_human_verification`
- `paused_for_login`

### 原则 5：先做 HTTP，后包 MCP

最小落地顺序建议：

1. 本地 HTTP API 跑通
2. Agent 宿主可直接调试
3. 稳定后再包成 MCP 工具

原因：

- HTTP 最快验证
- 调试门槛最低
- 后续 MCP 只是包装层，不改核心执行逻辑

---

## 6. 本项目中的能力映射

按当前项目能力，未来本地执行网关应优先映射这些动作：

### 已有或接近已有的能力

- 底部输入框设置文本
- Send 按钮发送
- Sync 按钮把双方最新回复写入对方输入框
- 抓取左右最新回复
- 判断某个 provider 是否仍在回复中
- provider 切换
- 界面布局切换

### 需要补强后再对外暴露的能力

- 明确的全局状态查询
- 当前 pane 与 provider 的稳定映射
- 人机验证 / 登录失效检测
- 高风险执行确认
- 单会话占用锁

---

## 7. 状态机建议

建议统一维护以下状态：

- `idle`
- `draft_ready`
- `sending`
- `waiting_reply`
- `sync_ready`
- `syncing_draft`
- `requires_human_verification`
- `paused_for_login`
- `awaiting_user_confirmation`
- `error`

### 状态含义

`idle`

- 当前没有待执行动作

`draft_ready`

- 底部输入框已有文本，但尚未发送

`sending`

- 正在把输入内容发送到目标 pane

`waiting_reply`

- 至少一个 provider 仍在输出中

`sync_ready`

- 双方都已回复完成，可以执行同步草稿

`syncing_draft`

- 正在把对方最新结论写入另一侧输入框

`requires_human_verification`

- 页面检测到验证码、二次校验或反自动化页面

`paused_for_login`

- 页面已退出登录，或停在登录流程中

`awaiting_user_confirmation`

- 某个高风险动作必须等待用户明确确认

`error`

- 当前流程发生本地执行错误，不能继续自动推进

---

## 8. 最小接口设计

以下先以本地 HTTP API 为例。后续若改成 MCP，保留同样的语义即可。

所有接口都只监听：

- `127.0.0.1`

并要求：

- 启动时生成随机 token
- 每次请求带 token
- 一次只允许一个控制会话

### 8.1 `GET /status`

用途：

- 获取当前执行状态
- 获取 pane / provider 映射
- 获取是否可发送、是否可同步
- 获取是否卡在验证 / 登录

返回建议：

```json
{
  "ok": true,
  "state": "sync_ready",
  "busy": false,
  "canSend": true,
  "canSync": true,
  "requiresHumanVerification": false,
  "pausedForLogin": false,
  "panes": [
    { "id": "pane-1", "provider": "chatgpt", "busy": false },
    { "id": "pane-2", "provider": "gemini", "busy": false }
  ]
}
```

### 8.2 `POST /input/set`

用途：

- 设置底部公共输入框文本

请求建议：

```json
{
  "text": "请分别分析机器人三大定律的现实问题。"
}
```

### 8.3 `POST /send`

用途：

- 触发一次发送
- 行为等价于点击当前底部 `Send`

请求建议：

```json
{
  "target": "all"
}
```

可扩展为：

- `all`
- `pane-1`
- `pane-2`

### 8.4 `POST /discussion/sync-draft`

用途：

- 抓取左右两侧最新稳定回复
- 按讨论模板交叉写入对方输入框
- 只写入，不自动发送

这个接口必须保持与当前产品原则一致：

- Sync 只同步
- 不发送

### 8.5 `GET /discussion/latest`

用途：

- 读取当前左右两侧最新回复
- 供 Agent 做判断、摘要、下一步规划

返回建议：

```json
{
  "ok": true,
  "replies": [
    {
      "paneId": "pane-1",
      "provider": "chatgpt",
      "busy": false,
      "text": "..."
    },
    {
      "paneId": "pane-2",
      "provider": "gemini",
      "busy": false,
      "text": "..."
    }
  ]
}
```

### 8.6 `POST /confirm`

用途：

- 用户确认高风险动作后，由 Agent 或本地界面显式提交确认

典型场景：

- 即将执行非讨论型网页动作
- 即将跳转敏感页面
- 即将发起用户不一定预期的操作

### 8.7 `POST /session/claim`

用途：

- 抢占当前控制会话
- 防止多个 Agent 同时控制同一个本地客户端

### 8.8 `POST /session/release`

用途：

- 释放控制会话

---

## 9. 不建议暴露的接口

以下能力不建议直接开放给 Agent：

- 任意执行 JavaScript
- 任意 DOM selector 点击
- 任意 URL 跳转
- 任意 provider 内部按钮坐标控制
- 直接清理所有 session / cookie

理由：

- 容易破坏当前登录态
- 容易让状态机失真
- 难以调试
- 难以保证安全

---

## 10. 人机验证与登录失效处理

这是后续必须明确实现的一层能力，而不是“出问题再说”。

### 10.1 进入验证状态时的行为

一旦检测到以下任意情况：

- 验证码
- Cloudflare / 风控挑战页
- 二次验证页面
- 登录失效
- 账号异常提示

系统应立刻：

1. 停止自动发送
2. 停止自动同步
3. 保留当前轮上下文
4. 状态切换为 `requires_human_verification` 或 `paused_for_login`
5. 向本地界面和 API 一起暴露该状态

### 10.2 恢复方式

建议恢复流程：

1. 用户在本机网页中手动完成验证 / 登录
2. 用户点击“继续”或 Agent 再次调用恢复接口
3. 系统重新检查状态
4. 回到 `idle`、`draft_ready` 或 `sync_ready`

关键要求：

- 不要整轮重来
- 不要丢失当前待发文本
- 不要丢失上一轮已抓取结果

---

## 11. 安全边界

虽然是本地接口，也不能裸奔。

最低限度要有：

### 11.1 本地监听

只监听：

- `127.0.0.1`

不要默认开放到局域网。

### 11.2 随机鉴权 token

启动时生成随机 token，保存在当前运行实例中。

### 11.3 单控制会话

同一时间只允许一个 Agent 控制会话。

### 11.4 危险动作确认

即使是本地 Agent，也不应该默认拥有无限控制权。

### 11.5 审计日志

至少记录：

- 谁占用了会话
- 调用了哪个接口
- 参数摘要
- 返回结果
- 出错原因

---

## 12. 推荐实现顺序

### 阶段 1：先把内部状态收口

目标：

- 让 SchemeChat 自己先能稳定返回统一状态

需要做：

- 增加全局状态管理
- 汇总各 pane 的 busy / latest reply / verification 状态
- 明确 `canSend` / `canSync`

### 阶段 2：抽象本地控制服务

目标：

- 把现有 UI 按钮背后的行为抽成内部 service

需要做：

- `setInputText`
- `send`
- `syncDraft`
- `getLatestReplies`
- `getStatus`

这一步完成后，UI 和 API 都调用同一套内部能力。

### 阶段 3：加本地 HTTP API

目标：

- 先用最直接的方式让本机 Agent 能调用

需要做：

- 监听 `127.0.0.1`
- token 鉴权
- 单会话锁
- 最小 5 到 8 个接口

### 阶段 4：接入 Agent 宿主

目标：

- 用 Codex / OpenClaw 一类本机 agent 跑通一条真实链路

最小链路：

1. Agent 设置输入
2. Agent 发送
3. Agent 轮询等待回复完成
4. Agent 拉取双方最新消息
5. Agent 触发 Sync 草稿
6. Agent 再决定是否发送下一轮

### 阶段 5：再考虑 MCP 包装

目标：

- 面向更多本地 Agent 宿主复用

说明：

- MCP 是接入层，不是核心执行层
- 不要一开始就把精力花在协议包装上

---

## 13. 最小验收标准

当以下条件都满足时，说明“本地 Agent 执行网关”第一版成立：

1. 本机 Agent 宿主可以通过本地接口设置输入文本。
2. 本机 Agent 宿主可以触发一次发送。
3. 本机 Agent 宿主可以读取当前 pane 状态与最新回复。
4. 当双方回复完成后，Agent 可以触发一次 Sync 草稿。
5. Sync 只同步，不自动发送。
6. 一旦遇到验证码或登录失效，系统会进入明确暂停状态。
7. 用户处理完成后，流程可以继续，而不是整轮报废。
8. 同一时间不会被两个 Agent 同时抢控制。

---

## 14. 当前不做的事

第一版明确不做：

- 自建云端调度服务
- 远程浏览器托管
- 公网开放本地接口
- 任意网页自动化控制
- 复杂权限系统
- 多用户协同控制
- 自动绕过验证码

---

## 15. 一句话结论

这个项目最合理的演进方向，不是“把网页搬到云上给 Agent 控”，而是：

**继续让网页留在用户本机，把 SchemeChat 做成一个受控的本地网页执行器，再给本机 Agent 宿主开放一层高层语义 API。**

这条路最现实，也最容易真正落地。
