# SchemeChat 统一产品方向

## 目的

这份文档用于替代此前分散的增长方向讨论，收束为一份新的 source of truth，明确：

1. SchemeChat 下一阶段最值得做的统一方向是什么。
2. 三个高价值方向应该怎样融合，而不是做成三个割裂功能。
3. 界面和能力应该怎样保持简单，但结果足够强。
4. 怎样同时兼容开发者用户和普通用户，而不把产品做散。

---

## 一句话定义

SchemeChat 不再继续往“多网页聊天壳”扩张，而是明确收束为：

`面向任务与问题的 AI War Room`

它的核心不是“同时打开多个 AI”，而是：

- 围绕一个 repo / issue / PR / spec / 任务书 / 现实问题
- 让多个网页 AI 进行多轮讨论
- 让 Codex / Claude Code 作为执行者或信息提供者接入
- 自动沉淀为 Markdown 结果物
- 让结果可以回放、分享、复用

这里的关键是：

`开发任务是最强场景，但不是唯一场景。`

同一个闭环也可以服务普通用户，例如：

- 对比几个方案该怎么选
- 让多个 AI 帮忙拆解一个复杂问题
- 先出方案，再让其他 AI 复核
- 把一场讨论整理成清晰结论

---

## 三个方向的统一关系

此前讨论出的三个高价值方向，本质上不该做成三个并列产品，而应是同一个产品的三层结构。

### 方向一：AI War Room

这是主场景，也是产品的核心入口。

用户带着一个真实任务进来，例如：

- 看一个仓库应该怎么改
- 拆一个 GitHub issue
- 审一个 PR
- 让多个 AI 对一份 spec 收敛方案
- 让 Codex / Claude Code 根据讨论结果执行
- 或者让多个 AI 帮普通用户比较方案、压缩分歧、形成结论

这一层回答的是：

`为什么开发者会打开 SchemeChat。`

### 方向二：Replay / Export / Share

这不是独立主功能，而是 War Room 跑完后的自然结果层。

一次讨论结束后，系统应自然产出：

- 结论摘要
- 分歧压缩
- 执行计划
- 风险清单
- 可分享 Markdown
- 可回放讨论记录

这一层回答的是：

`为什么这场讨论值得保存、传播、发到 GitHub。`

### 方向三：Role / Skill / Agent Gallery

这也不应单独先做成大平台，而应作为前两层稳定后的复用层。

当某些角色配置、讨论套路、任务模板反复证明有效，就把它们沉淀成：

- role preset
- task preset
- skill
- agent template

这一层回答的是：

`为什么用户会持续回来，并愿意贡献模板和玩法。`

---

## 正确的融合方式

正确的产品结构不是：

- 一个 War Room 页面
- 一个 Export 页面
- 一个 Gallery 页面

而应该是：

`一个主入口 + 一个结果页 + 一个模板库`

也就是：

1. 用户从一个简单入口发起任务
2. War Room 负责讨论和协作闭环
3. 结果页负责把产出整理成可读、可发、可复用的 Markdown artifact
4. 模板库只沉淀已经被证明有效的流程和角色

这样三者是前后关系，不是并列关系。

---

## 应该长成什么样

### 入口要简单

界面不应该长出很多按钮、很多参数、很多面板模式。

更合理的是只保留少数几个用户真正能理解的输入：

- 任务是什么
- 这次是讨论、执行还是评审
- 需要快一点还是深一点
- 是否让 Codex / Claude Code 参与

用户不应该被迫理解：

- pane 编排细节
- 轮次 prompt 结构
- 材料包拼接逻辑
- MCP 低层工具名

这些都应该由系统在内部处理。

### 能力要很强

虽然入口简单，但内部应该自动完成完整闭环：

1. 根据任务类型生成合适的讨论草稿
2. 选择轮次数量和讨论强度
3. 让多个网页 AI 分角色推进
4. 必要时调用 Codex / Claude Code 获取代码、测试、实现信息
5. 把这些信息重新注入讨论流
6. 最终生成一份高质量 Markdown 结果物

也就是：

`前端少选择，后端多编排。`

## 建议的统一产品模型

### 两层表达，同一个内核

为了兼容开发者和普通用户，产品表达应分成两层，但底层能力保持同一套：

- 对开发者：
  - 讲 repo / issue / PR / spec / agent 协作
- 对普通用户：
  - 讲分析问题、生成方案、检查结果

也就是说：

- 首页和首次上手文案要偏通用
- README、演示视频、GitHub 传播可以偏开发者
- 内部仍然走同一套讨论、收束、导出闭环

这样不会让产品分裂成两个版本。

### 一个主产品名词

建议后续对外统一讲述为：

`SchemeChat AI War Room`

不要对外讲成三个产品。

### 三种任务类型

建议只暴露三种高层任务类型，足够覆盖大多数场景：

#### 1. Explore

用于：

- 理解仓库
- 分析 issue
- 收敛方案
- 做技术判断
- 理清复杂问题
- 对比多个选项

产出重点：

- 问题定义
- 方案对比
- 风险点
- 建议路线

#### 2. Execute

用于：

- 让 Codex / Claude Code 接任务
- 让网页 AI 做执行前审议
- 输出任务书、补丁建议、测试建议
- 生成可执行方案
- 拆解下一步行动

产出重点：

- 执行计划
- 改动范围
- 验收标准
- 实施风险

#### 3. Review

用于：

- 审查 PR
- 审查实现方案
- 审查生成结果
- 复核结论
- 找漏洞、矛盾和遗漏

产出重点：

- 关键问题
- 风险优先级
- 是否建议合并
- 后续修复建议

这三类已经足够强，而且用户容易理解。

对普通用户展示时，可以进一步转成更直白的中文：

- Explore = 分析问题
- Execute = 生成方案
- Review = 检查结果

对开发者则保留英文任务类型，方便和工程场景对齐。

### 三档讨论强度

保留已有的 3 / 4 / 5 轮能力，但对外只讲成简单强度：

- Quick
- Standard
- Deep

内部再映射到现有轮次模式。

### 两类执行者

外部 agent 不需要做成一大堆入口，只需抽象成两类可选参与者：

- Codex
- Claude Code

它们在产品里扮演的角色不是“主聊天窗口”，而是：

- 代码信息提供者
- 执行者
- 验证者

## 最关键的结果物

为了让功能简单但价值高，所有路径都应该尽量收束到少数几种 Markdown artifact。

建议优先支持：

### 1. Decision Brief

适合 Explore。

内容包括：

- 背景
- 候选方案
- 共识
- 关键分歧
- 建议结论

### 2. Execution Plan

适合 Execute。

内容包括：

- 目标
- 任务拆解
- 文件范围
- 风险
- 验收标准

### 3. Review Memo

适合 Review。

内容包括：

- 关键发现
- 严重级别
- 建议修复
- 结论

### 4. Replay Snapshot

适合作为附加导出。

内容包括：

- 轮次摘要
- 关键引用
- 最终结论

只要这几种 artifact 做好，很多“分享、传播、复盘、复用”能力都会自然长出来。

对普通用户来说，这些结果物也同样成立，因为本质上都是把多 AI 的讨论收束成可读成果，而不是让用户自己翻长对话。

## 为什么这样更容易获得 GitHub stars

因为它同时满足了三个传播条件：

### 一眼能懂

开发者一看就知道这是：

`让多个 AI 围绕真实开发任务协作的工作台`

### 一眼有差异

它不是普通聚合聊天工具，也不是单模型 agent 壳，而是：

- 多网页 AI
- 多轮讨论
- 外部执行者接入
- Markdown 结果物沉淀

### 一眼能演示

很容易做出 30 秒演示：

1. 贴一个 issue 或仓库问题
2. 跑一轮 AI War Room
3. 让 Codex / Claude Code 补充实现或验证
4. 导出一份结论 Markdown

这比“再加一个按钮”更容易被 star。

补充一点：

GitHub stars 的传播主故事仍然应该偏开发者，因为开发者更容易理解 repo / issue / PR / agent 协作的价值，也更容易转发和二创。

但产品本身不应只允许开发者使用，而应让普通用户也能直接进入同一工作流。

---

## 不该做成什么样

为了避免产品重新发散，下面这些方向当前都不应该作为主界面重点：

- 大量 provider 切换按钮
- 大量手动 pane 编排开关
- 大量 prompt 高级参数
- 独立的大型 Gallery 市场首页
- 与主链路无关的低层 MCP 调试入口

这些能力可以存在，但不应抢主叙事。

---

## 推荐的实施顺序

### 阶段 1：收束成一个主故事

完成目标：

- 对外统一叫 AI War Room
- 明确三种任务类型
- 明确三档讨论强度
- 明确结果物以 Markdown 为主

### 阶段 2：把结果物做实

完成目标：

- Decision Brief
- Execution Plan
- Review Memo
- Replay Snapshot

### 阶段 3：让外部 agent 接入主闭环

完成目标：

- Codex / Claude Code 能作为执行者或验证者进入讨论
- 外部信息能回注到最终结果物

### 阶段 4：再沉淀模板

完成目标：

- role preset
- task preset
- skill / agent template

只有前面三步站稳，第四步才不会变成空平台。

---

## 当前 source of truth

从本文件开始，SchemeChat 的产品方向统一收束为下面这套判断：

1. 主方向不是继续堆聊天壳功能，而是 `AI War Room`。
2. Replay / Export 不是平行功能，而是 War Room 的结果层。
3. Gallery 不是当前主入口，而是成熟流程的沉淀层。
4. 前端必须继续保持简单入口，复杂编排放到系统内部。
5. 默认产出应尽量收束为 Markdown artifact，而不是更多零散 UI 状态。
6. 开发者是传播主场景，但普通用户也必须能使用同一套内核。
7. 产品应采用“双层表达”：外部传播偏开发者，实际入口兼容普通用户。

---

## 一句话版本

`用一个简单入口，把多 AI 讨论、外部 agent 执行、结果导出和模板沉淀串成一个闭环。`
